Code Validation

The way we validated and checked our code, both HTML and CSS, was by using W3C validator, which is a code validator. This site has an option where you can validate your code by direct input, and that is what we did.

Accessibility


We also made some coding choices with universal design and accessibility in mind. Firstly we decided to use W3’s Web Content Accessibility Guidelines and fulfill some criterias. The principles of web accessibility is represented in the acronym POUR The first letter, “P”, stands for perceivable and refers to perceivable information and user interface. A criteria we fulfilled in this section would be criteria 1.1.1 non-text content. By adding alternate text for non-text content, we fulfilled this criteria. By doing this, a screen reader can read out a text explaining the content for blind people.

The “O” stands for operable and refers to operable user interface and navigation. One criteria we fulfilled under this section is criteria 2.4.2 page title, as we provided descriptive titles for our web pages. We also fulfilled criteria 2.4.4 link purpose, as the link texts we use describe the purpose of the links. Both these criterias help the users understand where certain content is and what it is about.

The “U” stands for understandable and refers to understandable information and user interface. One criteria we fulfilled is 3.1.1 language of page, as we used the “lang” element in HTML to set the language to english. This helps the screen reader to convert text into synthetic speech.

The last letter, “R”, stands for robust and refers to robust content and reliable interpretation. We fulfilled criteria 4.1.1 parsing, by validating our code as mentioned above. By validating our code we ensure that our code is written properly so that assistive technologies can parse the content accurately and without crashing.

We also used a powerpoint provided by Yavuz Inal, associate professor at NTNU, to look for guidelines. The guidelines provided in the powerpoint and that we followed include good captioning for images and large click areas. We also used WebAims contrast checker to check if the colors we used are acceptable in relation to W3’s Web Content Accessibility Guidelines. By doing this we fulfill criteria 1.4.3, which ensures that people with low vision are able to read the text and content.
